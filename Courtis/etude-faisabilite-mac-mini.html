Résumé exécutif

Contexte

DAP utilise actuellement un VPS Hostinger pour héberger ses automatisations n8n (voice bot, pipelines Telavox, boîte à idées, etc.). L'idée est d'installer un Mac Mini en local comme serveur pour :

  • Héberger nos outils internes (n8n, bases de données, applications web)
  • Faire tourner des modèles d'IA en local (LLM) pour le traitement de données, l'analyse de documents et l'assistance aux courtiers
  • Garder les données sensibles des clients en interne (conformité RGPD / FSMA simplifiée)
  • Réduire les coûts récurrents d'hébergement à long terme
Conclusion : Le projet est faisable et recommandé. Un Mac Mini M4 Pro (24 Go RAM / 512 Go SSD) est capable de faire tourner l'ensemble de nos outils actuels + des modèles LLM performants en local. L'investissement initial est d'environ 1 849 € (matériel + accessoires), avec un coût récurrent de seulement ~15 €/mois. L'installation peut être réalisée en interne en suivant ce guide, sans compétences avancées en administration système.

Installation et configuration du Mac Mini comme serveur

1.1 — Choix du matériel

Le Mac Mini M4 (2024) est la meilleure option prix/performance pour notre usage. Voici la comparaison des configurations :

Configuration Puce RAM SSD Prix Verdict
Mac Mini M4 base M4 (10 CPU / 10 GPU) 16 Go 256 Go 699 € Limité pour LLM
Mac Mini M4 16/512 M4 (10 CPU / 10 GPU) 16 Go 512 Go 929 € OK sans LLM
Mac Mini M4 Pro 24/512 M4 Pro (12 CPU / 16 GPU) 24 Go 512 Go 1 649 € Recommandé
Mac Mini M4 Pro 24/1TB M4 Pro (12 CPU / 16 GPU) 24 Go 1 To 1 879 € Confort stockage
Mac Mini M4 Pro 48/512 M4 Pro (14 CPU / 20 GPU) 48 Go 512 Go 2 119 € LLM avancés
Pourquoi le M4 Pro 24 Go ? La RAM est le facteur critique pour les LLM. Avec 24 Go de mémoire unifiée (partagée CPU/GPU), on peut faire tourner des modèles de 7 à 14 milliards de paramètres de façon fluide. Le M4 base avec 16 Go est trop juste si on veut combiner n8n + PostgreSQL + LLM en simultané. Le 48 Go serait du luxe pour notre usage actuel.

1.2 — Paramétrage initial du Mac Mini (pas à pas)

Voici les étapes exactes à suivre dès le déballage :

1

Configuration macOS initiale

Lors du premier démarrage, créer un compte administrateur dédié (ex : dap-admin). Connecter le Mac au réseau filaire via Ethernet (port intégré) pour une connexion stable.

  • Activer le chiffrement FileVault (Réglages Système → Confidentialité et sécurité → FileVault) — protège les données en cas de vol
  • Désactiver la mise en veille : Réglages Système → Économie d'énergie → "Empêcher la mise en veille automatique quand l'écran est éteint"
  • Activer le démarrage automatique après coupure : Réglages Système → Économie d'énergie → "Démarrer automatiquement après une panne de courant"
2

Activer l'accès distant (SSH + Screen Sharing)

On aura rarement besoin de brancher un écran au Mac Mini. Tout se fera à distance :

  • SSH : Réglages Système → Général → Partage → Session à distance (cocher) — permet de se connecter en ligne de commande depuis n'importe quel PC du bureau
  • Partage d'écran : même menu → Partage d'écran (cocher) — permet de voir le bureau du Mac à distance via VNC
  • Pare-feu : Réglages Système → Réseau → Pare-feu → Activer le pare-feu (les connexions entrantes SSH et VNC seront autorisées automatiquement)
# Tester la connexion SSH depuis un autre PC du bureau : ssh dap-admin@192.168.1.XX # remplacer par l'IP du Mac Mini # Vérifier que le Mac est bien accessible : ping 192.168.1.XX
3

Fixer l'adresse IP locale

Le Mac Mini doit toujours avoir la même IP sur le réseau local pour que les services restent accessibles :

  • Réglages Système → Réseau → Ethernet → Détails → TCP/IP
  • Passer en Manuel (DHCP avec adresse manuelle)
  • Choisir une IP hors de la plage DHCP du routeur (ex : 192.168.1.200)
  • Masque : 255.255.255.0 — Passerelle : IP du routeur (souvent 192.168.1.1)
Alternative : Configurer une réservation DHCP dans le routeur (associer l'adresse MAC du Mac Mini à une IP fixe). C'est plus propre si on a accès à l'interface du routeur.
4

Installer Homebrew (gestionnaire de paquets)

Homebrew est l'outil standard pour installer des logiciels en ligne de commande sur macOS. C'est la fondation de tout le reste.

# Installer Homebrew (une seule commande) : /bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)" # Vérifier l'installation : brew --version
5

Installer Docker Desktop

Docker permet de faire tourner n8n, PostgreSQL et d'autres services dans des conteneurs isolés. C'est la méthode la plus propre et la plus facile à maintenir.

# Installer Docker Desktop via Homebrew : brew install --cask docker # Lancer Docker Desktop (une seule fois, il démarrera ensuite automatiquement) : open /Applications/Docker.app # Vérifier : docker --version docker compose version
Docker Desktop est gratuit pour les entreprises de moins de 250 employés et/ou moins de 10 M$ de CA. DAP est donc parfaitement éligible.
6

Configurer Cloudflare Tunnel (accès externe sécurisé)

Cloudflare Tunnel permet d'exposer les services du Mac Mini sur internet sans ouvrir aucun port sur le routeur et sans IP fixe publique. C'est la solution la plus sécurisée et la plus simple.

# 1. Installer cloudflared : brew install cloudflared # 2. Se connecter au compte Cloudflare : cloudflared tunnel login # → Un lien s'ouvrira dans le navigateur pour autoriser le tunnel # 3. Créer un tunnel : cloudflared tunnel create dap-server # 4. Configurer les routes (ex : n8n accessible sur n8n.dap-assurances.be) : cloudflared tunnel route dns dap-server n8n.dap-assurances.be # 5. Créer le fichier de config ~/.cloudflared/config.yml : tunnel: <TUNNEL_ID> credentials-file: /Users/dap-admin/.cloudflared/<TUNNEL_ID>.json ingress: - hostname: n8n.dap-assurances.be service: http://localhost:5678 - hostname: apps.dap-assurances.be service: http://localhost:3000 - service: http_status:404 # 6. Installer comme service (démarre au boot) : sudo cloudflared service install
Avantages de Cloudflare Tunnel :
  • Gratuit (plan Free suffit)
  • HTTPS automatique (certificats gérés par Cloudflare)
  • Aucun port ouvert sur le routeur = surface d'attaque minimale
  • Protection DDoS incluse
  • Pas besoin d'IP fixe publique

Implémentation des outils et LLM

2.1 — Déploiement de n8n et des outils internes

Tout sera orchestré via un fichier docker-compose.yml unique. Voici la configuration complète :

# Fichier : /opt/dap-server/docker-compose.yml # À créer dans le dossier /opt/dap-server/ version: "3.8" services: # --- Base de données --- postgres: image: postgres:16 restart: always environment: POSTGRES_DB: n8n POSTGRES_USER: dap POSTGRES_PASSWORD: [MOT_DE_PASSE_FORT] volumes: - postgres_data:/var/lib/postgresql/data ports: - "5432:5432" # --- n8n (orchestration) --- n8n: image: n8nio/n8n:latest restart: always depends_on: - postgres environment: DB_TYPE: postgresdb DB_POSTGRESDB_HOST: postgres DB_POSTGRESDB_DATABASE: n8n DB_POSTGRESDB_USER: dap DB_POSTGRESDB_PASSWORD: [MOT_DE_PASSE_FORT] N8N_HOST: n8n.dap-assurances.be N8N_PROTOCOL: https WEBHOOK_URL: https://n8n.dap-assurances.be/ volumes: - n8n_data:/home/node/.n8n ports: - "5678:5678" # --- Uptime Kuma (monitoring) --- uptime-kuma: image: louislam/uptime-kuma:1 restart: always volumes: - uptime_data:/app/data ports: - "3001:3001" volumes: postgres_data: n8n_data: uptime_data:
# Lancer tous les services : cd /opt/dap-server docker compose up -d # Vérifier que tout tourne : docker compose ps # Voir les logs en temps réel : docker compose logs -f
Migration depuis Hostinger : n8n permet d'exporter tous les workflows en JSON (Paramètres → Exporter). Il suffira d'exporter depuis l'instance Hostinger et d'importer sur la nouvelle instance locale. Les credentials (API keys, etc.) devront être re-saisis manuellement pour des raisons de sécurité.

2.2 — LLM en local : frameworks et modèles

Le framework recommandé pour faire tourner des LLM sur Mac est Ollama. Il est spécialement optimisé pour Apple Silicon et utilise la mémoire unifiée GPU/CPU du M4 Pro.

# Installer Ollama : brew install ollama # Lancer le serveur Ollama (tourne en arrière-plan) : ollama serve # Télécharger un modèle (ex : Llama 3.1 8B) : ollama pull llama3.1:8b # Tester en ligne de commande : ollama run llama3.1:8b "Résume en 3 points les avantages d'une assurance RC professionnelle" # L'API REST est disponible sur http://localhost:11434 # Compatible avec le format OpenAI (drop-in replacement)

Modèles recommandés pour DAP (M4 Pro 24 Go)

Modèle Taille RAM requise Vitesse (M4 Pro) Cas d'usage DAP Qualité
Llama 3.1 8B 4.7 Go ~8 Go ~40 tokens/s Résumés, classification, FAQ, analyse de mails Excellent
Mistral 7B 4.1 Go ~7 Go ~45 tokens/s Idem, très bon en français Excellent
Gemma 2 9B 5.4 Go ~9 Go ~35 tokens/s Analyse de documents, extraction d'infos Très bon
Llama 3.1 70B (Q4) 40 Go ~44 Go Trop lent — Incompatible
Phi-3 Mini 3.8B 2.2 Go ~4 Go ~60 tokens/s Classification rapide, tâches simples Bon (rapide)
Qwen 2.5 14B 8.7 Go ~14 Go ~20 tokens/s Tâches complexes, raisonnement Très bon
Recommandation : Commencer avec Llama 3.1 8B et Mistral 7B. Ces deux modèles sont excellents en français, rapides (~40 tokens/s = réponse quasi-instantanée), et laissent assez de RAM pour n8n + PostgreSQL. Ils couvrent 90% de nos besoins : résumés d'appels, classification de mails, extraction d'informations de documents, FAQ automatisée.
Limites : Les modèles 7-14B sont performants mais ne rivalisent pas avec Claude Sonnet/Opus ou GPT-4 pour des tâches très complexes (raisonnement juridique poussé, rédaction créative longue). Pour ces cas-là, on continuera à utiliser les API cloud (Claude API). L'approche hybride est la meilleure stratégie : LLM local pour le volume (tâches répétitives, données sensibles) + API cloud pour la qualité (tâches complexes ponctuelles).

2.3 — Intégration n8n + Ollama

n8n peut appeler Ollama directement via son noeud HTTP Request ou via le noeud AI natif :

// Exemple d'appel Ollama depuis n8n (HTTP Request Node) // URL : http://host.docker.internal:11434/api/generate // Méthode : POST // Body : { "model": "llama3.1:8b", "prompt": "Résume cet appel téléphonique : {{$json.transcription}}", "stream": false } // Depuis n8n 1.30+, le noeud "Ollama Chat Model" est disponible nativement // Configuration : Base URL = http://host.docker.internal:11434

Cas d'usage concrets pour DAP

  • Résumé automatique des appels Telavox — Retranscription → LLM local → résumé structuré → email au courtier
  • Classification des emails entrants — Nouveau sinistre / Demande de devis / Suivi de dossier → routage automatique
  • Extraction d'informations de documents — Lecture de polices, attestations, relevés → remplissage automatique Brio (quand l'API sera disponible)
  • FAQ interne améliorée — Le voice bot peut interroger le LLM local au lieu de Claude API pour les questions simples (coût = 0€)
  • Analyse de sentiment — Détecter les clients mécontents dans les transcriptions pour alerter le management

2.4 — Version pilote (test minimaliste)

Avant de tout migrer, on peut tester une version minimale en quelques heures :

1

Installer Docker + n8n + Ollama sur le Mac Mini

Suivre les étapes 4-5 de la section 1.2 + installer Ollama. Durée : ~30 minutes.

2

Créer un workflow de test

Un workflow simple : webhook → Ollama (résumé) → email. Tester avec un texte d'appel fictif. Durée : ~1 heure.

3

Comparer les résultats

Exécuter le même workflow avec Claude API et avec Ollama. Comparer qualité et vitesse. Durée : ~30 minutes.

4

Valider la stabilité

Laisser tourner pendant 48h. Vérifier les logs, la consommation RAM, la température. Si OK → feu vert pour la migration complète.

3.2 — Sauvegardes

Stratégie de sauvegarde 3-2-1 : 3 copies, 2 supports différents, 1 copie hors site.

Type de sauvegarde Quoi Où Fréquence Comment
Time Machine Système complet Disque externe USB Automatique (1x/heure) Brancher un disque externe, activer Time Machine
Backup Docker volumes n8n, PostgreSQL, configs Cloud (Backblaze B2 ou S3) 1x/jour (nuit) Script cron + rclone (voir ci-dessous)
Export n8n workflows Tous les workflows JSON OneDrive DAP 1x/semaine Workflow n8n automatisé
Dump PostgreSQL Base de données complète Cloud + disque local 1x/jour pg_dump automatisé
# Script de backup quotidien (/opt/dap-server/backup.sh) #!/bin/bash DATE=$(date +%Y-%m-%d) BACKUP_DIR="/opt/dap-server/backups/$DATE" mkdir -p $BACKUP_DIR # 1. Dump PostgreSQL docker exec dap-server-postgres-1 \ pg_dump -U dap n8n > "$BACKUP_DIR/postgres-$DATE.sql" # 2. Copier les volumes Docker docker run --rm \ -v n8n_data:/source:ro \ -v $BACKUP_DIR:/backup \ alpine tar czf /backup/n8n-data-$DATE.tar.gz -C /source . # 3. Sync vers le cloud (Backblaze B2) rclone sync $BACKUP_DIR b2:dap-backups/$DATE # 4. Nettoyer les backups de plus de 30 jours find /opt/dap-server/backups -maxdepth 1 -mtime +30 -exec rm -rf {} \; # Planifier avec cron : crontab -e # 0 2 * * * /opt/dap-server/backup.sh

Risques et solutions de secours

5.1 — Matrice des risques

⚡ Coupure de courant Moyen

Impact : Arrêt brutal du serveur, corruption potentielle des données.

✅ Mitigation : Onduleur (UPS) qui maintient le courant ~15 min, laissant le temps d'un arrêt propre. macOS redémarre automatiquement après coupure. PostgreSQL est résistant aux crashs (journalisation WAL).

🌐 Coupure internet Moyen

Impact : Services inaccessibles de l'extérieur. Les webhooks Twilio ne fonctionnent plus.

✅ Mitigation : Routeur 4G backup avec basculement automatique. Les services locaux (LLM, n8n en réseau local) restent accessibles. Cloudflare Tunnel se reconnecte automatiquement.

🔧 Panne hardware Faible

Impact : Serveur hors service pendant la durée de la réparation/remplacement.

✅ Mitigation : Apple offre une garantie de 1 an (extensible à 3 ans avec AppleCare+ pour ~100€). En cas de panne : restaurer les backups cloud sur un VPS temporaire en quelques heures. Les Mac Mini sont réputés très fiables (pas de pièces mobiles sauf le ventilateur).

🔒 Sécurité / intrusion Faible

Impact : Accès non autorisé aux données clients.

✅ Mitigation : Cloudflare Tunnel = aucun port ouvert sur le routeur. FileVault = disque chiffré. Pare-feu macOS actif. Accès SSH par clé uniquement (désactiver l'authentification par mot de passe). Mises à jour de sécurité régulières.

🏢 Vol / dégât physique Faible

Impact : Perte du serveur et des données locales.

✅ Mitigation : FileVault rend les données illisibles sans mot de passe. Backups cloud quotidiens = données récupérables. Assurance bureau couvre le matériel. Option : câble antivol Kensington (~20€).

📋 Perte de données Faible

Impact : Workflows, configurations ou données clients perdues.

✅ Mitigation : Stratégie backup 3-2-1 (Time Machine local + Backblaze cloud + export workflows OneDrive). Test de restauration trimestriel. RPO (perte max) = 24h. RTO (temps de restauration) = ~4h sur un VPS temporaire.

5.2 — Plan B : solution hybride

En cas de problème prolongé avec le Mac Mini, voici le plan de secours :

FONCTIONNEMENT NORMAL : Internet ─── Cloudflare Tunnel ─── Mac Mini (local) ├── n8n ├── PostgreSQL ├── Ollama (LLM) └── Uptime Kuma EN CAS DE PANNE (Plan B) : Internet ─── Cloudflare DNS ─── VPS Hostinger (fallback) │ ├── n8n (restauré depuis backup) │ └── PostgreSQL (restauré depuis dump) │ └── Bascule DNS en ~5 min via Cloudflare dashboard POUR LES LLM (si Mac Mini HS) : Workflow n8n ─── Claude API (Haiku 4.5) ─── Fallback cloud Coût : ~0,001€/requête Qualité : supérieure au LLM local
Avantage de l'approche hybride : On conçoit les workflows n8n pour basculer automatiquement entre Ollama (local) et Claude API (cloud). Si le Mac Mini est HS, les workflows critiques (voice bot, pipeline Telavox) continuent de fonctionner via l'API cloud, avec un coût marginal. Aucune interruption de service pour les clients.

Conclusion et avis

Le Mac Mini M4 Pro est une solution solide, économique et parfaitement adaptée aux besoins actuels et futurs de DAP. Voici les arguments clés :

  • Faisable techniquement — Tous les logiciels nécessaires sont gratuits et bien documentés. L'installation peut être réalisée en interne.
  • Rentable financièrement — Investissement de ~1 849 €, amorti en moins d'un an. Économie de ~15 000 € sur 5 ans vs cloud équivalent.
  • Sécurisé — Données en local (RGPD), chiffrement, aucun port ouvert, backups automatisés.
  • Évolutif — On commence avec n8n + LLM simples, et on peut ajouter des cas d'usage progressivement.
  • Réversible — En cas de problème, on bascule sur le cloud en quelques minutes grâce à l'architecture hybride.

Prochaine étape recommandée : présenter ce rapport à Christophe, valider le budget, et commander le matériel. Une démonstration pilote peut être réalisée en amont sur un Mac existant si souhaité.
